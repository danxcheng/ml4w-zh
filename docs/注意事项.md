# ml4w-zh 注意事项与技术细节

## 为什么需要这个工具？

ML4W dotfiles 是持续更新的（rolling release），每次更新会覆盖用户对配置文件的修改。传统的做法有：

1. **`custom.lua`** — 官方推荐的自定义入口，但对快捷键绑定无效（hl.bind 先注册优先匹配）
2. **`PROTECTED` 文件** — 在文件夹里放空文件 `PROTECTED`，安装器跳过整个目录
3. **`hook.sh`** — 在 `~/.ml4w-hyprland/hook.sh` 里用 `_protect` 保护特定文件

这个工具结合了方案 2 和 3：自动建 PROTECTED 文件，配合 YAML 翻译定义，ml4w 更新后跑一次 `ml4w-zh check` 就知道哪些要重新应用。

## 技术要点

### 1. Hyprland Lua 配置的快捷键优先级

Hyprland 的 `hl.bind()` 是**先注册的优先匹配**。`custom.lua` 加载晚于 `conf/keybindings/default.lua`，所以 **custom.lua 里用 hl.bind 无法覆盖默认快捷键**。

解决方案：直接修改源文件（`conf/keybindings/default.lua`），这个工具就是这样做的。配合备份机制，更新后重新应用即可。

### 2. PROTECTED 机制

ml4w 安装器在复制文件时，会检查目标目录下有没有名为 `PROTECTED` 的空文件。如果有，**跳过整个目录**不覆盖。

```bash
touch ~/.config/hypr/scripts/PROTECTED
```

这样 `scripts/` 下的所有自定义文件（包括汉化后的脚本）都不会被 ml4w 更新覆盖。

### 3. ml4w/scripts/ 没有 PROTECTED

`~/.config/ml4w/scripts/` 是 ml4w 管理的脚本目录，ml4w 安装器可能覆盖它。如果需要保护，在 `~/.ml4w-hyprland/hook.sh` 里加：

```bash
_protect .config/ml4w/scripts/ml4w-wallpaper-automation
_protect .config/ml4w/scripts/ml4w-install-system-updates
```

### 4. Waybar JSON 的限制

`modules.json` 是 JSON 格式，但 ml4w 原版文件里**可能包含尾逗号**（trailing comma）。标准 JSON 解析器会报错，但 Waybar 的 C++ 解析器容忍。手动编辑时注意保持格式一致。

### 5. QML 应用的汉化

Quickshell 应用（欢迎页、侧边栏、日历、壁纸等）使用 QML（Qt Markup Language）。汉化方式：

- 用 `qsTr()` 包裹的字符串 → 改 `qsTr("...")` 里的内容
- 硬编码的字符串 → 直接改 `text: "..."` 或字符串常量
- 部分 QML 用 JavaScript 动态生成文字 → 在 JS 函数里改

### 6. GitHub 上的原始 bug

ml4w 原版 WelcomeApp 的 "Hyprland Homepage"、"Hyprland Wiki"、"Update ML4W OS" 三个菜单项都错误指向 `https://github.com/mylinuxforwork/wallpaper`。这个 bug 需要手动修复。

## 自动构建与 CI

### 方案一：手动模式（推荐）

```bash
# ml4w 更新后
cd ~/项目/ml4w-zh
ml4w-zh check      # 看哪些文件被覆盖了
ml4w-zh apply      # 重新应用汉化
```

### 方案二：GitHub Actions 自动检测

在 `.github/workflows/check-translations.yml` 中配置：

- 定时（比如每周）检查 ml4w 最新 release
- 下载最新版配置文件
- 用 YAML 翻译定义测试补丁能否应用
- 如果有冲突，自动提 Issue 通知

### 方案三：本地 git 对比

把 ml4w 源文件作为一个 git submodule，汉化补丁作为 patch 文件。ml4w 更新后，用 `git rebase` 思路重新应用 patch，冲突时手动解决。
